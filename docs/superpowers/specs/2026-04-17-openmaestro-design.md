# OpenMaestro — Design Specification
**Date:** 2026-04-17  
**Status:** Approved for implementation  
**Author:** OA Maestro  
**Target ship:** Tonight (v1)

---

## What It Is

OpenMaestro is an open-source, LLM-agnostic browser automation framework. Clone the repo, run one setup command, then tell your AI what you want to automate in a browser — and it does it, on your machine, in your browser, using tools you already pay for.

**One-line positioning:**  
> "Give any AI hands. Automate anything you do in a browser — on your machine, with your subscriptions, using whatever AI you already have."

---

## What It Is NOT

- Not a new subscription or SaaS
- Not limited to Claude Code (works with Cursor, Windsurf, GitHub Copilot, more)
- Not limited to Higgsfield (works with any browser-based tool)
- Not a cloud service (runs entirely on the user's machine)
- Not a GUI app (terminal + browser window, that's it)
- Not trying to be OpenClaw — it's the trust-first local alternative

---

## The Core Insight

Most people who want browser automation face a binary: pay for a cloud agent service (OpenClaw, Manus, CoWork) and hand over your credentials, or figure out Playwright yourself. OpenMaestro is neither. It packages the intelligence layer (what the AI knows about browser automation) and the inspection layer (how the AI understands a tool before scripting it) into a repo you clone and own.

The "already logged in" insight is the differentiator: the user's browser is already authenticated to every tool they pay for. Playwright operates in that browser session. No credential sharing ever.

## Setup Script (`install.sh` / `install.ps1`)

Single interactive script. Does exactly these steps in order:

**Step 1 — Detect AI**  
Prompts user:
```
Which AI are you using?
  [1] Claude Code
  [2] Cursor  
  [3] Windsurf
  [4] GitHub Copilot (VS Code)
  [5] Other — I'll configure manually
```

**Step 2 — Verify Node.js**  
If missing: prints install link (nodejs.org), exits cleanly with message. Does NOT attempt to install Node — that's one step too many to do silently.

**Step 3 — Install Playwright + stealth stack** (local to repo)
```bash
npm install playwright playwright-extra puppeteer-extra-plugin-stealth
npx playwright install chromium
```

**Step 4 — Configure Playwright MCP for chosen AI**

| AI | Action |
|---|---|
| Claude Code | `claude mcp add playwright npx @playwright/mcp@latest --headed` |
| Cursor | Write to `.cursor/mcp.json` |
| Windsurf | Write to `.windsurf/mcp.json` |
| VS Code Copilot | Write to `.vscode/mcp.json` |

MCP config always includes `--headed` flag (browser is visible — this is the trust value prop) and stealth browser args:
```json
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": [
        "@playwright/mcp@latest",
        "--headed",
        "--browser", "chrome",
        "--launch-option", "--disable-blink-features=AutomationControlled",
        "--launch-option", "--window-size=1920,1080"
      ]
    }
  }
}
```

**Step 5 — Write platform brain file**  
Copies `OPENMAESTRO.md` to the appropriate filename for the chosen AI.

**Step 6 — Health check**  
Verifies Playwright can launch a browser. Prints pass/fail.

**Step 7 — Done message**
```
OpenMaestro is ready.

Open this folder in [chosen AI] and say:
"Read your instructions and tell me what you can do."

Your AI now has a browser. Tell it what to automate.
```

## The Brain File (`OPENMAESTRO.md`)

This is the product. Six sections:

### Section 1 — Identity & Capabilities

Tells the AI what it is and what Playwright MCP gives it:
- Navigate to any URL
- Click any element (by selector, text, role)
- Type text (including `slowly: true` for human-like input)
- Take screenshots (full page or element)
- Read/extract DOM content
- Execute JavaScript on the page
- Wait for elements to appear
- Scroll, hover, drag
- Handle file uploads and downloads
- Fill forms, select dropdowns

Also states: stealth mode is active, browser is visible to user, user can type "stop" at any time to halt.

### Section 2 — Onboarding Protocol

When a user wants to automate something new, run this before touching anything:

1. "What tool or website do you want me to work in?" → get URL
2. "Walk me through what you do manually, step by step — be as specific as possible."
3. "Are there any steps where you make a decision or judgement, or is every step the same every time?"
4. "What does success look like? How do you know the task is done correctly?"
5. "Are there any things I should never do, never click, or never change?"
6. "Should I work while you watch, or can I run this unattended?"
7. "Is there anything about how YOU specifically do this that's different from how most people would?" (surfaces personal style, quality bars, business rules)

If Video Vision MCP is available (check via `/mcp`): offer user the option to share a screen recording of their workflow instead of step 2-3. Analyse the recording to extract steps and decisions.

After onboarding: save to workflow spec before touching browser.

### Section 3 — Research + Inspection Protocol

After onboarding, before scripting, always:

**Research phase (web search):**
1. Search "[tool name] automation known issues"
2. Search "[tool name] Playwright OR Selenium known selectors OR quirks"
3. Search "[tool name] bot detection OR anti-automation"
4. Note any keyboard shortcuts that are more reliable than clicking
5. Add findings to workflow spec "Research Notes" section

**Inspection phase (inspect.js):**
1. Call `node scripts/inspect.js <URL>` — returns structured JSON
2. Read the `interactive_elements` array to identify actual selectors
3. Read `potential_blockers` — if bot detection signals present, warn user before proceeding
4. Read the screenshot path — review it to verify the page loaded as expected
5. If login page detected: stop and tell user to log in manually first (never attempt login)
6. Add selector map to workflow spec "Observed Selectors" section

**Bot detection warning language:**
> "This site has [cloudflare/hcaptcha/datadome] anti-bot systems. Stealth mode is active but cannot guarantee success. Shall I proceed?"

### Section 4 — Workflow Spec Generation

After research + inspection, call `node scripts/scaffold-workflow.js <workflow-name>` then fill in every section:
- Onboarding answers
- Research notes
- Observed selectors
- Step-by-step automation script
- Known issues and workarounds
- Resume state table (one row per step)

Save to `workflows/[name].md`. Confirm with user before proceeding to execution.

On subsequent runs: read the saved workflow spec — no need to re-run onboarding.

### Section 5 — Execution Rules

- Always clear contenteditable fields via JS before typing (not just Playwright's type)
- Use `slowly: true` for all typing on platforms with bot detection
- Take a screenshot after each major step to verify state
- Wait for network idle after navigation before interacting
- Never store, display, or log passwords or API keys
- User must be logged in before starting — verify on first screenshot
- Update resume state in `workflows/[name]-session.md` after each completed step

### Section 6 — Error Handling + Recovery

| Situation | Action |
|---|---|
| Element not found | Screenshot → wait 3s → retry once → if still missing, stop and ask |
| Page didn't load | Wait for network idle → retry navigation → if failing, stop and report |
| CAPTCHA detected | Stop immediately, tell user to solve it, resume after confirmation |
| Unexpected modal/popup | Screenshot → attempt to dismiss → if cannot, stop and ask |
| Session crashed | User says "resume [workflow]" → read `workflows/[name]-session.md` → continue from last completed step |
| Rate limit message | Stop → tell user → suggest wait time → resume on confirmation |

## `scripts/inspect.js`

**What it does:** Opens a stealth Playwright browser, navigates to a URL, extracts structured page info, saves a screenshot, outputs JSON to stdout.

**How the AI calls it:**
```bash
node scripts/inspect.js https://higgsfield.ai/image/nano_banana_2
```

**Output schema:**
```json
{
  "url": "string",
  "title": "string",
  "screenshot": "./tmp/inspect-[timestamp].png",
  "interactive_elements": [
    {
      "type": "button|input|select|textarea|contenteditable",
      "text": "string (button label)",
      "selector": "string (CSS or data-attr selector)",
      "aria_label": "string",
      "input_type": "string (for inputs)",
      "placeholder": "string (for inputs)",
      "options": ["array (for selects)"],
      "current_value": "string",
      "visible": true,
      "position": { "x": 0, "y": 0 }
    }
  ],
  "potential_blockers": {
    "login_required": false,
    "captcha_detected": false,
    "captcha_type": "none|hcaptcha|recaptcha|turnstile",
    "bot_detection_signals": [],
    "rate_limit_message": false
  },
  "notes": "string (any anomalies observed)"
}
```

**Implementation:** Uses `playwright-extra` + `puppeteer-extra-plugin-stealth` (installed locally by setup script). Self-contained Node.js script, no external dependencies beyond local npm install.

**Error handling:** URL not reachable → JSON with error field. Auth redirect → sets `login_required: true`. Timeout (15s) → returns partial results with timeout flag.

## `scripts/scaffold-workflow.js`

**How the AI calls it:**
```bash
node scripts/scaffold-workflow.js my-higgsfield-batch
```

**Creates:** `workflows/my-higgsfield-batch.md` and `workflows/my-higgsfield-batch-session.md`

The workflow `.md` template has these sections (AI fills them in):
- Metadata (created date, target URL, AI used)
- What this workflow does
- Onboarding answers
- Research notes
- Observed selectors map
- Automation steps (numbered, with selector references)
- Known issues and workarounds
- Notes

The session `.md` is a resume state table updated by the AI as steps complete:
```markdown
# Session: my-higgsfield-batch
Last updated: [timestamp]
Current status: in_progress | completed | paused

| Step | Description | Status | Notes |
|------|-------------|--------|-------|
| 1 | Navigate to Higgsfield | ✅ Done | |
| 2 | Confirm settings | ✅ Done | |
| 3 | Generate prompt batch | 🔄 In progress | On prompt 3/10 |
```

Recovery command: "Resume my-higgsfield-batch" → AI reads session file → continues.

## Workflow Templates (v1 Examples)

Three examples ship with v1. All are runnable markdown — the AI can follow them directly.

### `higgsfield-content-batch.md`
Automates batch image/video generation on Higgsfield. Includes the documented JS clear fix for the contenteditable prompt bar. Supports any Higgsfield model (NanoBanana, Seedance, Cinema Studio, etc.) — the user specifies model URL in setup. Saves outputs to `output/YYYY-MM-DD/`.

### `linkedin-post-scheduler.md`
Given a list of posts and publish times, opens LinkedIn, navigates to post composer, writes each post, schedules it. Flagged as bot-detection risk — includes `slowly: true` on all typing, explicit stealth warning in file header.

### `research-compiler.md`
Given a list of URLs, visits each, extracts main content via DOM inspection, compiles into a single markdown document. Zero bot detection risk. Ideal first demo for new users — works reliably on any public website.

## Install Experience

**One command:**

Mac/Linux:
```bash
git clone https://github.com/oamaestro/openmaestro && cd openmaestro && bash install.sh
```

Windows (PowerShell):
```powershell
git clone https://github.com/oamaestro/openmaestro; cd openmaestro; .\install.ps1
```

Or paste into Claude Code / Cursor terminal — works identically.

**Post-traction (v1.1):** Publish to npm → `npx openmaestro`. ~2 hour job once repo has stars.

## Open Questions — Resolved

| Question | Decision | Reason |
|---|---|---|
| Windows: WSL or native PowerShell? | Native PowerShell | User is on Windows 11 Pro, WSL adds friction |
| `.mcp.json` committed or generated? | Generated by setup | Platform-specific paths differ; committing one would break others |
| Workflow specs: Markdown or JSON? | Markdown | Human-readable, editable by non-technical users, AI reads it naturally |
| Example workflows: runnable or docs? | Semi-structured runnable markdown | Structured enough for AI to follow, readable enough for humans to edit |
| Video Vision MCP: v1 or optional? | Optional hook in brain file | Separate project; hook ready for when it ships |
| Stealth browser: v1 or later? | v1, two layers | MCP launch args (basic) + playwright-extra in inspect.js (full) |
| GUI: include? | No | Terminal + visible browser IS the UI; GUI adds build time and contradicts philosophy |
| Scheduling/daemon: v1? | No | Headless background tasks are a different trust model; market as "you ignite it" |
| npx one-liner: v1? | No — git clone for tonight, npx for v1.1 | Avoids npm publishing overhead for tonight's ship |
| Name | OpenMaestro | "Open" = free/transparent/open-source; "Maestro" = orchestrator; connects to OA Maestro persona |
